# Essay Writing Style Guide

All essays are written in Japanese. This guide describes the style rules to follow.

## Tone & Voice

- Use polite `〜です / 〜ます` form as the base. Mix in `〜である / 〜だ` form situationally
- Polite but not stiff. Add occasional humor or self-deprecating remarks for warmth
- Avoid absolutist phrasing (`〜すべき`, `〜であるべき`). Prefer softer expressions that leave room for the reader to think (`〜と考えています`, `〜ではないでしょうか`)
- Actively use questions and empathy-seeking expressions to create a dialogic style
- Share personal experiences candidly — let the author's personality come through
- Use forward-looking, action-encouraging language
- At pivotal moments, use short, punchy sentences for impact
  - Example: `成果物だけであれば、コードを見れば分かります。しかし、エンジニアリングの本質は、制約の中でどのような意思決定を下したかというストーリーの中にこそ宿ります。`
- When expressing passion or conviction, don't shy away from literary or poetic phrasing
  - Examples: `コードに宿る暗黙知を言語化し`, `紡がれた無数の対話のログこそが、技術文化そのものです`, `未来への投資報告書`

## Structure

Structure patterns differ by publication target.

### Pattern A: Internal Communication (Intranet/Wiki)
- **Opening**: One sentence stating what inspired the essay (e.g., `『脳に収まるコードの書き方』の冒頭に非常に良い話があったので触発されて書いてみた`)
- **Purpose (目的)**: Bullet list of what the essay will explain
- **Target readers (想定読者)**: Explicitly state who this is for (e.g., `エンジニアが計画通りに進捗してくれないと感じているマネージャー`)
- **Summary (まとめ)**: Present conclusions/key points BEFORE the main body, so readers grasp the big picture first
- **Body**: Develop one argument per section
- **Closing (おわりに)**: Recap + call to action for readers

### Pattern B: External Engineering Blog
- **Opening**: Self-introduction (affiliation, role) → Frame the topic with passion (e.g., `私たちが組織として最も熱量を注ぎ、大切に育ててきた文化について紹介します`)
- **Definition**: Define the essence of the topic early (e.g., `私たちは技術力評価会を〜『対話の場』であると定義しています`)
- **Body**: Multi-angle exploration with concrete episodes and citations
- **Closing (おわりに)**: Short, with lingering resonance

### Common Rules
- Headings should be assertions or questions — never mere labels
  - Good: `屋根から作ることができる`, `「何をしたか」ではなく「なぜそれを選んだか」を語る`, `「自分が一番詳しい」を受け入れよう`
  - Avoid: `技術選定について`, `評価のポイント`
- One argument per section
- Walk the reader through the "why" — let them retrace the thought process

## Expression Techniques

### Metaphors
- Use metaphors to introduce complex concepts
- Always acknowledge the limits of the metaphor (e.g., `庭を育てる喩えはソフトウェアを作る活動を完全には表現できていません`)
- Leverage contrast structures (e.g., "building a house" vs "building software")

### Citations & Quotes
- Insert quotes from books or notable figures to add authority and depth
- Briefly introduce the source's background (e.g., `ウィキを開発したエンジニアでありデザインパターンやエクストリームプログラミングのパイオニアであるウォード・カニンガム`)
- Embed actual feedback or statements as block quotes for immediacy
- Weave links to relevant internal/external blog posts naturally into the text

### Concrete Examples
- After every abstract claim, provide at least one concrete episode or case
  - Example: Framework selection using a specific UI library → the evaluation forum asks `なぜ他の選択肢ではだめだったのか？`
- Use examples that help readers relate the topic to their own situation

### Other
- Minimize jargon. When essential, explain origins and context (e.g., tracing the conceptual roots of terms like "technical debt")
- Do not overuse diagrams or code snippets. Prioritize clear prose
- Emphasize key points with bold text or rhetorical questions

## Reader Consideration

- Aim for comprehension without prior knowledge
- Gauge the reader's knowledge level and add supplementary explanations as needed
- Embed references and links naturally to provide paths for deeper exploration

## Core Themes

- Engineering organizations, management, team building
- Technology selection, design philosophy (emphasizing underlying principles, decision criteria, lessons learned)
- Software development processes
- Personal growth, careers, learning methods
- Communication, feedback
- Evaluation systems, organizational culture design
- Industry trends, emerging technologies
